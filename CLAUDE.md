# AGENTS Instructions

This repository is a monorepo that contains a demo AI agent application of Ichimaru-ramen food delivery planning system.

# Python environment

Python is needed to build and run the demo system. After installation, run `source .venv/bin/activate` to activate the Python environment. During development and test of this deom Python 3.12.10 is used.

# Skills

Project agent skills live under `ai/skills/<skill-name>/` (each with a `SKILL.md`). This is a custom location and is not auto-discovered — read the relevant `SKILL.md` before running a skill.

- `retrieve-regional-population` — download 2020 Census prefecture population CSVs into `DATA/s01_raw/`, then extract and combine their rows into the UTF-8 TSV `DATA/s02_intermediate/regional_population.tsv`.
- `retrieve-regional-geoshapes` — download 2020 Census prefecture boundary shapefiles (ZIP) into `DATA/s01_raw/`, then extract each into `DATA/s02_intermediate/geoshape_<NN>/` (NN = 2-digit prefecture code).
- `retrieve-weather-station` — download the JMA AMeDAS station master (`amdmaster.index4`) into `DATA/s01_raw/weather-station.csv`, keep only active stations (End Date = 9999-99-99), and write a reduced UTF-8 TSV `DATA/s02_intermediate/weather-station.tsv`.
- `retrieve-weather-history` — download JMA daily weather history (from January three years ago up to the month two days before today — at least three full calendar years, computed from the system date, with the final month truncated to that day; all prefectures combined, one request per month) into `DATA/s01_raw/weather-history-all-<YYYY-MM-01>.csv`, then reshape each to a long-format UTF-8 TSV `DATA/s02_intermediate/weather_history_all_<YYYY-MM-01>.tsv` (都道府県 column distinguishes prefectures; falls back to per-prefecture files if the combined request is rejected). This is the long step (~12-30 min, depending on JMA server load).

`make base-data` runs all four skills above in sequence (the weather-history step is long, ~12-30 min depending on JMA server load).

- `synthesize-stores` — population-weighted sampling of Ichimaru's store network. For each prefecture and store count in `docs/profiles/Locations.md`, samples that many distinct 大字・町 (ooaza) with probability proportional to 2020 Census population, places a store at each ooaza's polygon centroid (from the geoshape shapefiles), names it `<都道府県名><市区町村名><大字・町名>店`, draws weekday/weekend sales baselines, and writes the UTF-8 TSV `DATA/s03_primary/store.tsv`. Reproducible via `synthetics/random_seed` in `config/config.yaml`. Depends on the `retrieve-regional-population` and `retrieve-regional-geoshapes` outputs.
- `synthesize-pois` — same population-weighted ooaza sampling as `synthesize-stores`, but for POIs, placing each at a **uniform random point inside the sampled ooaza polygon** (not the centroid) and sampling **with replacement** (an ooaza may recur). Driven by `config/config.yaml` (`synthetics/competitors/*`, `synthetics/home_buildings/*`, `time_horizon`), it writes `DATA/s03_primary/competitor.tsv` (competitor stores with weekday/weekend baselines and `open_date`/`end_date`, where `end_date = open_date + business_duration` or `9999-99-99` if still open) and `DATA/s03_primary/home_building.tsv` (residences with a `unit` count, `open_date`, and a permanent `9999-99-99` `end_date`). Names get a `_<seq>` suffix to stay unique. Same intermediate-data dependencies as `synthesize-stores`.
- `synthesize-events` — same population-weighted ooaza sampling as `synthesize-pois` (random point inside the ooaza polygon, with replacement, `_<seq>`-suffixed names), for people-gathering events. Driven by `config/config.yaml` (`synthetics/events/{numbers,people_range}`, `time_horizon/sales_history`), it writes `DATA/s03_primary/event.tsv` (`prefecture`, `event_name`, `latitude`, `longitude`, `people`, `event_date`). `event_date` is a weighted random date over the `sales_history` window with April/August/October days weighted 5×. Same intermediate-data dependencies as `synthesize-stores`.
- `synthesize-sales` — synthesize daily store sales (bowls) per `docs/synthetics/Sales.md`. For every store in `store.tsv` and every date in `time_horizon/sales_history`, builds a weekday/weekend baseline (Sat/Sun = weekend) and adds influences from nearby home buildings (≤500 m), competitors (≤50 m), events (≤200 m), and the nearest active, temperature-capable weather station's temperature/rain (precipitation-only stations without a temperature sensor are excluded before matching), then rounds and clamps to ≥10. Writes `DATA/s03_primary/sales.tsv` (`prefecture`, `store_name`, `date`, `sales`). Reproducible via `synthetics/random_seed`. Depends on the `synthesize-{stores,pois,events}` and `retrieve-weather-{station,history}` outputs.
- `match-store-weather-station` — match each store to its nearest usable JMA weather station per `docs/synthetics/Match-store-weather-station.md`. Picks the nearest **active**, **temperature-capable** (`Temperature` flag = 1), georeferenced station that also appears in the weather history, by haversine distance; **stations without a temperature sensor (precipitation-only rain gauges) are removed before matching** so every store's station can report temperature. Persists the mapping to `DATA/s03_primary/matched_store_weather_station.tsv` (`prefecture`, `store_name`, store coords, `station_number`, `station_name` = the `観測地点` join key, station coords, `distance_m`). Uses the same station criteria as `synthesize-sales` and `dfm-create-features` (active + temperature-capable + georeferenced + observed), so each store's feature weather and actual weather come from the identical station. Stdlib only; deterministic. Depends on the `synthesize-stores`, `retrieve-weather-station`, and `retrieve-weather-history` outputs. `make synthetics` runs this skill as its final step.
- `dfm-create-features` — build the Demand Forecast Model feature data sets per `docs/demand-forecast/dfm_features.md`. One row per `(store_name, reference_date, target_date)` (reference_date = a Thursday; target_date = the 7 days of week+1), with regression variables (past-zone sales aggregates: week-1 and week-1..4 mean/median over all/weekday/weekend, week-1-minus-week-4 deltas, per-ISO-weekday means), weather variables (high/avg temperature for the target date via a forecast-unavailable previous-year proxy from the nearest active, temperature-capable station — precipitation-only stations excluded — applied identically in training and prediction; `week+1_rainfall` always 0 — the one-year lag carries no rainfall signal), calendar variables (`month_number`, `is_weekend`, `weekday_number`, cyclical `target_offdays_{cos,sin}`), and the `actual_sales` target (blank for prediction). Writes `DATA/s04_feature/{training,test,predict}_dataset.tsv`. Stdlib only; deterministic for a fixed JST date (overridable with `--today`). Depends on the `synthesize-{stores,sales}` and `retrieve-weather-{station,history}` outputs.
- `dfm-build-model` — train, tune, and evaluate the Demand Forecast Model per `docs/demand-forecast/dfm_modeling.md`. Fits a LightGBM regressor (scikit-learn API, `LGBMRegressor`) on `DATA/s04_feature/training_dataset.tsv`: drops the key columns (`store_name` is not a feature), pops `actual_sales` as the label, and grid-searches `max_depth`×`learning_rate`×`n_estimators` (45 combos) on a single time-ordered 6:4 split by `reference_date`, selecting the lowest validation MAPE. Refits the best params on the full training set, evaluates on `test_dataset.tsv`, and writes `DATA/s05_model/`: `dfm_lgbm.txt` (native booster text format) plus `model_parameters.json` (tuned params, seed, ordered `feature_columns` inference contract), `model_validation_metrics.json`, `model_test_metrics.json`, `test_scatter.png` (actual-vs-predicted scatter with the y=x line and R²), and `shap_beeswarm.png` (SHAP beeswarm of the refit model on the test set). Reproducible via `modeling/training/random_seed` in `config/config.yaml`. Does **not** score the prediction set (a separate skill). Depends on the `dfm-create-features` output. `make modeling` runs `dfm-create-features` then this skill.
- `dfm-predict-sales` — score the trained model on the prediction set per `docs/demand-forecast/dfm_prediction.md`. Loads the booster `DATA/s05_model/dfm_lgbm.txt`, reindexes `DATA/s04_feature/predict_dataset.tsv` to the saved `feature_columns` order (the inference contract; `Booster.predict` aligns by position), coerces features to float (blank→NaN), predicts week+1 sales, joins `prefecture` from `DATA/s03_primary/store.tsv` by `store_name`, and writes `DATA/s06_prediction/predicted_sales.tsv` (`prefecture`, `store_name`, `reference_date`, `target_date`, `predicted_sales` as a raw float — no rounding/clamping). Depends on the `dfm-build-model` and `synthesize-stores` outputs. `make prediction` runs this skill.
- `dfm-explain-predictions` — explain each prediction as per-feature SHAP contributions per `docs/demand-forecast/dfm_explanation.md`. Builds the feature matrix per the inference contract (drop the 3 key columns + `actual_sales`, reindex to `feature_columns`, blank→NaN), computes exact tree SHAP via LightGBM native `booster.predict(pred_contrib=True)` (per-feature contributions + base value; additive in bowls under the L2 objective), joins `prefecture` + `predicted_sales` from `DATA/s06_prediction/predicted_sales.tsv`, self-checks `base_value + Σcontribs ≈ predicted_sales`, and writes `DATA/s06_prediction/shap_values.tsv` (wide: `prefecture`, `store_name`, `reference_date`, `target_date`, `predicted_sales`, `base_value`, `shap_<feature>`×30) and `shap_values_long.tsv` (tidy: keys, `predicted_sales`, `base_value`, `feature`, `feature_value`, `shap_value`). Uses `lightgbm`+`pandas`+`numpy` (no `shap` package); deterministic. Depends on the `dfm-predict-sales`, `dfm-build-model`, and `dfm-create-features` outputs. `make prediction` runs this skill after `dfm-predict-sales`.
- `calibration-backtest` — back-test the trained model against known actuals per `docs/calibration/Backtest.md`. Re-scores the booster `DATA/s05_model/dfm_lgbm.txt` over the 4 most-recent distinct `reference_date` values of `DATA/s04_feature/test_dataset.tsv` (where `actual_sales` is populated), following the same `dfm_prediction.md` inference contract as `dfm-predict-sales` (drop key columns + `actual_sales`, reindex to saved `feature_columns`, blank→NaN), joins `prefecture` from `DATA/s03_primary/store.tsv`, and writes `DATA/s07_calibration/backtest_sales.tsv` (`prefecture`, `store_name`, `reference_date`, `target_date`, `actual_sales`, `predicted_sales`) with predicted and actual side by side. `actual_sales` is carried through unchanged (never fed to the model). This is the single source of the back-test window and the prediction source for the residuals/calibration analysis. Depends on the `dfm-build-model` and `synthesize-stores` outputs. `make calibration` runs this skill (then residuals).
- `calibration-calculate-residuals` — compute back-test residuals with model feature weather vs actual weather side by side per `docs/calibration/Residuals.md`. Reads `DATA/s07_calibration/backtest_sales.tsv` (adds `residual` = `predicted_sales` − `actual_sales`), joins the model's feature weather from `DATA/s04_feature/test_dataset.tsv` on the 3 key columns (`feature_` prefix; `feature_week+1_rainfall` is uniformly 0 by construction), and joins the actual weather from `DATA/s02_intermediate/weather_history_*.tsv` on the store's matched station (`観測地点`) and `target_date` (`日付`) via `matched_store_weather_station.tsv` (`actual_` prefix). Writes `DATA/s07_calibration/residuals.tsv`. Uses `pandas`; deterministic. Depends on the `calibration-backtest`, `match-store-weather-station`, and `dfm-create-features` outputs plus the weather history. `make calibration` runs this skill after the back-test.

# Python dependencies

Skills that need third-party libraries declare them in `requirements.txt` (kept lean). Install with `pip install -r requirements.txt` inside the activated `.venv`. Currently: `pyshp` (shapefile reading in the `synthesize-*` location skills), `PyYAML` (config reading in the `synthesize-*` skills, incl. `synthesize-sales`), and `lightgbm` + `pandas` + `numpy` + `scikit-learn` + `shap` + `matplotlib` (the `dfm-build-model` modeling skill).
